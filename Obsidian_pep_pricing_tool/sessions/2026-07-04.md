## Slow-query triage (export + root-cause fix)

User asked to work through the 408 rows in `pc_slow_query_cache` (all `status='new'`), offering a choice between building an export tool or directly investigating/fixing; chose both.

- **Export**: `GET /admin/slow-queries/export` (CSV, admin-only), "Export CSV" button on the System tab. Verified live.
- **Real finding**: the ranked-by-time list was mostly noise — the collector's own maintenance queries pinned at 12.4s by a stale one-time `GREATEST()`-latched spike, plus a harmless `SLEEP()` heartbeat query. Underneath that, 91 of 408 rows (90% of all logged occurrences — 122,077 of 135,441) were all `findExactProductMatch()` (`backend/lib/price_import.php`), which wrapped `canonical_name`/`alias` in `LOWER()`, defeating their UNIQUE indexes even though both columns already use case-insensitive collation (`utf8mb4_unicode_ci`) — a plain `=` already matches case-insensitively. Confirmed via `EXPLAIN` (full index scan → `const` lookup) and a real PHP call on the server (case-insensitive match still correct, non-match still returns `null`). Fixed by removing the `LOWER()` wrapping, deployed, verified.
- **Triage**: marked all 408 rows directly via SQL (per explicit instruction to get them from the DB and mark them, not via the admin PATCH API) — 94 `resolved` (the fix + noise entries, each with an explanatory `status_note`) and 314 `acknowledged` (reviewed, proportionate to real usage, already indexed at this dataset's size — no action needed). Verified live in the admin System tab.

## Raw/bulk powder pricing (user-surfaced new file shape, plus a same-day correction)

User uploaded a real vendor file (`scarlett_raw_2025.xls`) selling raw, unprocessed peptide powder priced per gram — no vial/kit concept, no Cat No, just compound name + flat per-gram price. Asked for a plan to support this kind of listing.

**First pass**: modeled as the same product entity (not a separate product per raw compound) with a new `1g` spec (`numeric_value=1000mg` — the schema already normalized grams to mg) and a new "Raw Material" classification tag, per the user's confirmed design choice. Added extraction-prompt rule 10 recognizing "$/G"-style listings. Imported the real file through the actual review-queue pipeline (created vendor "Scarlett," converted the source `.xls` to CSV since the upload endpoint doesn't accept the legacy format, processed it, approved all 57 resulting rows through the real Review Queue UI, bulk-tagged the resulting products via direct SQL).

Two real process issues hit and fixed along the way (not code bugs): a file-ownership permissions bug from placing the test upload via SSH instead of the real HTTP path (fixed with `chgrp`), and a rapid-click approval batch that likely double-submitted and froze a browser tab via a blocked `alert()` (recovered on a fresh tab with proper per-click confirmation instead of a fixed delay).

**Same-day correction**: user asked "how are these tagged as being raw?" — which surfaced that the classification tag was product-level, so filtering by "Raw Material" showed *every* spec of a tagged product (vial sizes included, not just the actual `1g` row) — confirmed live, 197 rows instead of the correct 57. Root-caused and fixed properly: migration `021_raw_material_spec.sql` moves the flag onto `pc_specifications.is_raw_material` (backfilled for the 57 existing raw specs), retires the classification tag entirely, and threads the real flag through `findOrCreateSpec()`, the extraction prompt (now emits `is_raw_material` directly instead of leaving it inferred), the Review Queue (new checkbox), and a real `rawMaterialOnly` filter in the Comparison query/both exports/query-log rerun. Frontend: a real "Raw/bulk powder only" checkbox replacing reliance on the classification pill, plus a "Raw" badge on the spec column. Verified live post-fix: toggling the filter returns exactly the 57 real rows, no vial specs mixed in.

## Files/System tab row-alignment bug (recurrence of an earlier bug class)

User reported the Files tab had "the same alignment issue" seen before on the Products tab. Confirmed identical root cause: `.actions { display: flex }` applied directly to a `<td>` overrides its default `table-cell` display, breaking row height/alignment. Fixed the same way — removed the flex, spaced buttons with `button + button { margin-left: 4px }` instead. Proactively grepped the whole frontend for the same pattern (a class used on a `<td>` that also sets `display: flex`) and found one more latent instance not yet reported: `SystemTab.vue`'s `.sq-actions` (the slow-queries Acknowledge/Resolve/Reopen buttons) — fixed identically before anyone hit it. Verified live via DOM geometry (`getBoundingClientRect` + computed `display`) on both tables: all cells now share identical `top`/`height` and `display: table-cell`.
