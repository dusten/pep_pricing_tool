# Session 2026-07-06

Weekly clipping scan ran first (no new files). Rest of the session split between root-causing a recurring duplicate-product bug and a run of Comparison-table/pricing correctness fixes.

## Root cause of backlog #28's duplicate-naming pattern

Traced the recurring "doubled alias in canonical_name" pattern (previously just flagged, not root-caused) to extraction prompt rule 7: it fed Claude the product+alias catalog as `"Name (alias1, alias2)"` strings and asked it to map variants onto them, so Claude sometimes echoed the alias-annotated display string back as `canonical_name` itself. Because `products/merge.php` keeps a merged-away name as a new alias on the winner, each recurrence fed a longer alias into the next snapshot, compounding across merges — caught via a visibly double-nested name from the 2026-07-06 Lucy import. Removed rule 7 entirely (redundant with the software's own alias matching) and the unused `is_new_product` field. Cleaned up the specific instance found, verified live.

A broader sweep then found the same bug had hit 26 products in that morning's Lucy import batch, not just one — merged via `migration_scripts/2026-07-06-merge_lucy_duplicates_batch.php`. Catalog 220 → 194.

## Admin Products page N+1 fix

Investigating "every merge/alias action feels slow" found `ProductsTab.vue`'s `load()` firing one request per product (194 sequential calls) just to populate alias/classification chips, re-triggered after every write action. Bulk-fetched aliases + classifications in 2 extra grouped queries inside `products/index.php`'s existing list response; per-row specs moved to on-demand `loadSpecs()`. Page load: 195 requests → 1.

## Error-message precedence bug

Raw SQL text (`SQLSTATE[23000]...`) was leaking to the Inventory tab on a real duplicate-tier collision. Root cause: `frontend/src/utils/api.js`'s `data.message || data.error` picked the raw diagnostic text over the intended friendly headline, backwards across 7+ endpoints that follow the `{error, message}` convention. Fixed the precedence, not the data — the underlying collision was a legitimate `uq_price` hit.
