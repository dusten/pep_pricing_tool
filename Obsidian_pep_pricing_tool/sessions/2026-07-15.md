Rewrote migration 035 to guard on `information_schema.tables` so the same schema.sql-vs-migration collision can't recur on a fresh install either.

## Bell-curve gate switched from 75% vendor coverage to absolute floor of 8

User noticed well-sampled items (Adamax at 11-12 vendors) weren't qualifying for the bell curve. Root cause: the 75% gate measured against *all* active vendors (25), but most vendors only carry a subset of the catalog, making "75% of the whole store" an unrealistic bar regardless of how well an item was actually sampled. Changed both the backend gate and the frontend's separately-duplicated client-side check to `MIN_VENDORS_FOR_DISTRIBUTION = 8`; removed the now-dead coverage-percent UI copy.

## Phase 1 browser verification + storage-dir bug (found live)

In-browser pass on the deployed Phase 1: `/suggest-vendor` renders (relationship radios, contacts, template download, PII note, gated gold nav item); first real submit **500'd** — Apache can't `mkdir` under `storage/` (755 ec2-user), so the runtime creation of `vendor_suggestions/` was never going to work; the same latent bug covered `quarantine/` (an actual malware hit would have 500'd too), and `deploy.sh`'s `rsync --delete` would have wiped both dirs on the next deploy. Fixed on the server (`ec2-user:apache` 770 + `httpd_sys_rw_content_t`) and permanently in `setup.sh`/`add-price-site.sh`/`deploy.sh` excludes. Re-verified end to end: template CSV → 201 → instant score **98/100** (2/3 matched, `NotARealPeptide` correctly unmatched) → admin Vendor Suggestions tab (relationship pill, score breakdown, extracted-price table) → rejected with admin note. Accept flow deliberately not exercised against production; EICAR untested while ClamAV stays disabled.

## Phase 2 cron blocker: ec2-user not in apache group

The build's own verification surfaced that PHP-FPM writes uploads as `apache:apache` 750/644 while the cron runs as `ec2-user` — not in the `apache` group, so the worker couldn't read any *real* submission (test only passed because the file was re-chowned). Same gap sat dormant in Phase 1's >2MB async PDF path. Fixed live (`usermod -aG apache ec2-user`, verified fresh-session reads) and in both provisioning scripts. Phases 1–2 now genuinely operable; only Phase 3 (un-gate at launch + tier decision) remains.

## Admin approval gate + duplicate-file block added (backlog #69, same day)

Product owner tightened the abuse model after Phase 2 shipped: non-template files must never reach Claude without an explicit admin approval, and a user must not be able to burn repeat Claude calls resubmitting the same file. Migration 037 added `awaiting_approval` to the status enum (MODIFY COLUMN, existing rows untouched) and a `content_hash` column + `(user_id, content_hash)` index. Submit endpoint now sha256-hashes the upload and 422s on a same-user non-rejected duplicate; non-template files land `awaiting_approval` instead of `pending_parse`, requiring a new admin `queue` action (dynamic route `admin/vendor-suggestions/{id}/(accept|reject|queue)`) before the unmodified cron loop will ever claim them. Frontend: "Awaiting review" status (no Claude mention to users), admin tab's "Send to Claude" button. Verified server-side: `php -l` clean, `DESCRIBE` confirms the new enum/column, manual cron run left an `awaiting_approval` row untouched, and `diagnostic_scripts/check_suggestion_approval_gate.php` (insert/assert/cleanup) confirmed both the dedup-hash guard and the cron-claim exclusion. Deploy + smoke clean.
