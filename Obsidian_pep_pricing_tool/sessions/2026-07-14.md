# Session 2026-07-14

The longest day of the run: a silent row-drop bug triggered a broad reprocess that itself created a multi-wave duplicate-data mishap (cleaned up across 5 separate waves through the rest of the day), plus Comparison UI redesign work, a slow-query-log regression fix, and the first cut of the admin activity dashboard. Full incident writeup: [[wiki/analyses/2026-07-14-incomplete-spec-drop-bug]]; duplicate list: [[wiki/analyses/2026-07-14-flagged-duplicate-listings-needing-research]].

## Silent row-drop bug for incomplete spec data (CUV-5)

User noticed a product mentioned in a vendor file's processing notes never showed up anywhere. Root cause: `vendor_file_processor.php` silently dropped any extracted row with a real name+price but an empty/zero spec — no pending-import row, no log, even though Claude had already flagged the ambiguity via its own warning field. Fixed to route these to the Review Queue instead (new `incomplete_spec` match type, migration 032). A read-only scan across every historical extraction found the true scope: 5 dropped-row instances across exactly 3 files, all recovered via the no-Claude-call Manual JSON path.

## Reprocess-all mishap and cleanup (multi-wave)

Chasing the bug above, a broad "reprocess every file's stored JSON" pass (37 files) created 318 mostly-duplicate pending rows, since the pending-creation logic only checks current catalog state, not review history — reprocessing re-queued rows that had already been approved or rejected once. Bulk-rejected all 318 after confirming zero were genuine new findings, then restored 28 the user was actively reviewing via a precise targeted undo.

The same mishap kept surfacing as new failure modes through the rest of the day, since some resurrected data went straight through the exact-match auto-commit path (bypassing the pending queue entirely, so the first cleanup pass never touched it):
- **Second wave** — garbled `canonical_name`s (Claude's alias-echo artifacts, pre-dating this session) got approved as 3 new duplicate products; merged back into their correct originals, 12 more garbled/duplicate pending rows rejected after individual verification against active prices.
- **Third wave** — 22 duplicate-SKU price rows (the reprocess re-extracted an already-committed file under a different SKU abbreviation for the same real listing, e.g. "BC5" vs "BP5" both meaning BPC-157 5mg) across 8 vendors, deactivated.
- **Fourth wave** — 8 resurrected NOVI OREA rows reintroducing the already-fixed MOQ-in-vials bug (rule 12, fixed 2026-07-06) as bogus bulk-tier prices that would have wrongly won the "lowest price" highlight; deactivated.
- **Fifth wave** — a full duplicate-vendor audit (broadened beyond same-price-only matching, 75 groups found vs. 24 previously) caught 62 more instances of the identical pattern, all deactivated. The remaining 13 didn't fit the reprocess-mishap shape — traced to genuine vendor-source ambiguities instead; 2 resolved with user approval, 11 filed to the wiki for future research.

Every deactivation pass was verified against a database-wide `pc_prices`/`pc_specifications` product_id consistency check (0 mismatches each time) and archived as a `migration_scripts/` script.

## `pc_import_slow_queries` event regression found and fixed

Evaluating the slow-query log before making further changes found the hourly event had been silently failing every run since 2026-07-11 — migration 026 had accidentally reintroduced a `DELETE FROM mysql.slow_log` statement MariaDB rejects outright (already discovered and removed once, months earlier, by migration 006). The event's INSERT still succeeded, so the table looked alive, but counts were being inflated by re-importing the same never-cleared rows hourly. Also confirmed no genuinely slow queries exist today (max 0.082s) — every row was flagged purely by the rows-examined filter. Fixed via migration 033 (restores the proven SELECT-only body), cache purged.
