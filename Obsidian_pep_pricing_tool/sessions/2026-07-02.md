## 2. "What happens when a file fails" surfaced a real transaction bug; the file-preview feature surfaced three layered ones — pushed to origin/main

- **Transaction-rollback gap, found by directly answering a design question rather than assuming the happy path.** User asked what happens on a processing failure. Tracing it end to end: failures *before* `processVendorFile()`'s `$pdo->beginTransaction()` (Claude API error, bad zip, missing file) already worked correctly — clean exception, `processing_status='failed'` recorded with the real message. Failures *inside* the transaction (a price value overflowing `DECIMAL(10,2)`, any DB constraint hit) did not: the transaction was left open, and `process.php`'s own "mark as failed" update then rode on that same doomed, never-committed transaction — silently discarded right along with it when the request ended, leaving the file stuck at `processing_status='processing'` forever with zero error message. Checked prod first: no files were currently stuck (every real failure this session happened on the *safe* side, before the transaction), so this hadn't bitten yet, just hadn't been hit. Fixed with a try/catch/rollback/rethrow wrapped around the transaction body in `vendor_file_processor.php`, verified with an isolated test against a scratch `pc_app_settings` key (proved the in-transaction write correctly rolled back while a separate post-catch write persisted) rather than trusting the code-reading alone.
- **File preview feature — asked for as one thing, needed three separate bugs fixed to actually work.** User wanted a "View" button on the Files tab, in-page (closing it shouldn't navigate away). Building it surfaced, in order:
  1. **The existing Download link had been silently broken the entire time.** `/api/files/{id}/download` requires the same `Authorization: Bearer` header every other endpoint does, but Download was a plain `<a href>` — browsers can't attach custom headers to plain navigation. Confirmed via `curl`: a bare unauthenticated GET returns `401`. Nobody had ever actually clicked it successfully. Fixed by switching both Download and the new View to fetch-with-auth-header → blob → object URL, rather than direct navigation.
  2. **`download.php` served images/zips as `application/octet-stream`** — its MIME map only had `pdf`/`xlsx`/`csv`. A blob's `.type` comes straight from the response's `Content-Type`, and a browser won't decode a blob URL as an `<img>` unless the declared type is `image/*`, even with perfectly valid JPEG bytes underneath. Fixed by deriving the real image MIME from the stored file's extension (same pattern already used in `vendor_file_processor.php`'s image branch).
  3. **CSP had no `img-src`/`frame-src` directive**, falling back to `default-src 'self'` — which doesn't permit `blob:` sources at all. This was the one that took real digging: after fixing #2, the blob's `.type` was confirmed correct (`image/jpeg`, real 342KB size, captured by monkey-patching `URL.createObjectURL` to intercept it without touching the auth token directly, which trips a safety guard) — yet `naturalWidth` was still `0`. No console error surfaced through the normal console-reading tool (CSP violations don't reliably route through regular `console.*` capture). Confirmed by pulling the raw `Content-Security-Policy` response header directly via `curl` instead of guessing. Fixed in three places kept deliberately in sync (`public/index.php`'s PHP-level header, `price.conf`, `price-standalone.conf`) — worth remembering Apache's `Header always set` *overrides* whatever PHP sets for the same header name, so the PHP-side fix alone would never have taken effect live; the Apache vhost push was the one that actually mattered, applied with the by-now-standard `httpd -t` before `systemctl reload` safety check.
  Verified all three fixes together against the real deployed site (not just re-reading code): image `naturalWidth: 1280`/`854` after the full chain, PDF's iframe correctly holding a `blob:` src, close behavior confirmed to leave the admin on the exact same Files-tab DOM state (no navigation).
- **Pushed to `origin/main`**: 4 commits (`40eea3a..130e803`).

**Standing lesson:** a feature that looks like "just add a button" can be gated behind several independent, previously-invisible bugs that only surface once something actually exercises the path end to end (auth delivery mechanism, MIME correctness, CSP policy) — each layer looked fine in isolation and each was verified with real evidence (a `curl` header, a monkey-patched blob capture, actual `naturalWidth`) rather than accepted on reasoning alone once the first fix didn't resolve the symptom.

## 3. File preview follow-up: PDF only filling ~150px, zip shouldn't get a View button

Two small, precisely-reported bugs from actually using §2's feature: "Viewing a PDF only uses the top 25% of the card" and zip shouldn't show a View button at all (no meaningful inline preview exists for it, per §2's own fallback message). Zip fix was a one-line `v-if`. PDF fix took two real attempts, both verified against the live site via `getBoundingClientRect()` fill-ratio checks rather than trusting the CSS reasoning alone:

1. **First attempt (wrong): flex percentage-height ambiguity.** `.view-body` is a flex container with `align-items: center`, which opts children out of stretch — so the iframe's `height: 100%` never resolved against a definite value. Fix: `position: relative` on `.view-body`, `position: absolute; inset: 0` on the iframe, sidestepping flex-percentage resolution by sizing against the positioned ancestor's box instead. Deployed, reverified live — still broken, iframe still exactly 150×300 (the CSS *was* correctly on the server and correctly loaded by the browser, confirmed both ways, so this wasn't a deploy/cache miss).
2. **Second attempt (right): `<iframe>` is a replaced element.** `top/right/bottom/left: 0` were all correctly computed (confirmed via `getComputedStyle`), yet `width`/`height` still resolved to `300px`/`150px` — the browser's own HTML-spec default size for an iframe with no explicit dimensions. Replaced elements (iframe, img, video, canvas) don't auto-size from opposing `inset` values the way a plain `div` would; `inset: 0` alone isn't sufficient, `width: 100%; height: 100%` has to be set explicitly too. Added both together, reverified: fill ratio `1.00` on both axes — but this turned out to only fix the *container box*, not what was actually wrong (§4).

**Standing lesson:** when a CSS fix doesn't resolve the symptom, that's a signal the diagnosis was wrong, not that the fix needs a nudge — `getComputedStyle()`/`getBoundingClientRect()` on the actual live element found the real cause (replaced-element sizing rules) in one step once tried, after a plausible-but-wrong flex-percentage theory had already been "fixed" and deployed without changing anything. This lesson held for the container box, but §4 shows it wasn't the whole story.

## 4. PDF preview, continued: the box was never the real problem — five more root causes, three of them genuinely different bugs, before it actually worked

§3's fix made the `<iframe>` box itself correctly fill the card (`getBoundingClientRect()` confirmed 1:1). User reported it was **still** showing only the top portion — proof that a correctly-sized *container* says nothing about what a replaced element's own internal renderer chooses to draw inside it. That distinction is what made this the longest single debugging thread of the project so far. Full arc, in the order things actually happened:

### 4a. Native Chrome PDF viewer abandoned — asked the user directly rather than guess a 4th CSS fix

Chrome's built-in PDF viewer running inside an `<iframe src="blob:...">` has a long, independently-documented history of not correctly participating in resize/layout — multiple Chromium bug reports and forum threads describe the exact symptom (content stuck in a small region, doesn't track the container). Confirmed this was the real shape of the problem via web research (not assumed) before proposing a fix, then asked the user directly (per [[memory/feedback_ask_open_decisions_directly]]) whether to keep patching CSS or switch approach entirely. User chose to rebuild with `pdfjs-dist`, rendering to a plain `<canvas>` this app fully controls the sizing of — sidesteps the native viewer's internal rendering entirely rather than fighting it further.

### 4b. Screenshots were completely unavailable this entire debugging arc

Every `computer` screenshot call failed with a CDP `Page.captureScreenshot` timeout, across at least 6 different fresh tabs, for the whole remainder of this thread. Verification had to happen entirely through `getBoundingClientRect()`, `getComputedStyle()`, `read_network_requests`, monkey-patching `URL.createObjectURL`, and eventually the user's own DevTools screenshots — which is exactly what ended up cracking the last two bugs (§4d, §4e), since some failure modes (a MIME-type header, a CSP violation, a console error) simply aren't visible to DOM-measurement-based verification at all. Worth remembering for next time image-based verification is unavailable: pull the specific signal that actually carries the failure (response headers, `securitypolicyviolation` events, console text) rather than only checking box geometry.

### 4c. pdf.js integration built against the real installed package, not recalled API shape

Installed `pdfjs-dist` (initially latest, 6.1.200) and, rather than write `getDocument()`/`getViewport()`/`.render()` calls from training-data memory, checked the actual shipped `.d.ts` files for exact parameter names (`getViewport({scale})`, `render({canvasContext, viewport})`, `PageViewport.width/height`) and grepped the real worker source for `new Worker(...)` to confirm the exact construction pdf.js uses internally, before writing anything that depended on it. This discipline is what made the later diagnosis (§4e) fast once symptoms pointed there — already knew precisely which internal call was failing rather than needing to first map the library's shape.

### 4d. Apache had no MIME mapping for `.mjs` — worker silently fell back to synchronous main-thread rendering, looking exactly like a frozen tab

First real deploy of the canvas-based version hung completely — even a trivial, synchronous, unrelated DOM query timed out at the tool level (45s), repeatedly, across multiple fresh tabs. That pointed at genuine main-thread blocking, not "still loading." `curl -I` on the worker's own URL showed why: `Content-Type: text/plain`. Apache's default `mime.types` has no `.mjs` entry, and combined with `X-Content-Type-Options: nosniff` (a header this project set earlier for unrelated reasons), the browser refuses to execute it as a module worker — real, but silent: no thrown error visible to the page, pdf.js just quietly falls back to its "fake worker" mode, which parses and renders the entire PDF synchronously on the main thread instead of a background worker. For any real PDF, that's slow enough to look identical to a hung tab. Fixed with `AddType text/javascript .mjs` in both `price.conf` and `price-standalone.conf`, applied live with the by-now-standard `httpd -t` before `systemctl reload` safety check (asked the user for explicit confirmation first, consistent with how every other live Apache push this project has gone).

### 4e. The user's own browser was still serving the pre-fix response — because only server config changed, not the file, so Vite's content hash never rotated

User re-tested, still broken, still the exact same `text/plain` error verbatim in their console — even though `curl` (from a completely different network path) now showed the correct header. Vite names build assets by content hash; since only the *Apache config* changed and not the worker file's actual bytes, the URL (`pdf.worker.min-DEtVeC4l.mjs`) stayed byte-identical across the fix, so any cache — the user's browser, or anything between them and origin — that had already stored the broken response for that exact URL had no reason to ever re-fetch it. Hard-refreshing doesn't reliably help either, since it's not necessarily the browser's own cache making the call. Fixed by appending a manual `?v=1` query string to the worker URL pdf.js requests, forcing a genuinely new cache key regardless of what any layer thought it already had. **Standing lesson:** a server-config-only fix to a content-hashed asset doesn't automatically bust caches the way a real content change would — don't assume a same-URL fix is live everywhere just because origin now serves it correctly.

### 4f. A real, upstream pdf.js bug: native `Uint8Array.prototype.toHex()` called with no feature detection, unsupported before Chromium 140

With the worker now loading correctly (confirmed: `Content-Type: text/javascript`, and — critically — only one clean request for the worker instead of a failed-then-fallback pair), a *new*, different error appeared: `"a.toHex is not a function"`. This looked at first like it might just be fallout from §4d/§4e cascading, but the user's own DevTools network panel proved otherwise: the worker's response headers were now genuinely correct and it loaded in a single clean request, yet the exact same error persisted — meaning this was a real, independent third bug, not a symptom of the first two.

Traced to pdf.js's own source (grepped the actual installed package, not assumed): its internal "fingerprints" computation (an internal per-document identifier, unrelated to page rendering) calls `hashOriginal.toHex()` directly with no existence check. `Uint8Array.prototype.toHex()` is a genuinely new JS engine method — confirmed via web research (a qutebrowser GitHub issue hitting the identical error, and a maintainer's comment linking [caniuse](https://caniuse.com/mdn-javascript_builtins_uint8array_tohex) confirming it landed in **Chromium 140**). The user's browser, confirmed via their own DevTools request headers screenshot (`User-Agent: ... Chrome/128.0.0.0 ...`), is on **Chrome 128** — a version that predates this method by a meaningful margin, so the native call is simply `undefined` and throws.

The same maintainer suggested pdf.js's "legacy" build as the fix. Checked directly rather than trusting the suggestion at face value: **the legacy build of pdfjs-dist 6.1.200 has the exact same unguarded `hashOriginal.toHex()` call** — "legacy" in this package apparently targets different module/syntax compatibility, not missing browser-native methods, so that specific suggestion turned out not to apply here. Instead, checked progressively older published versions' actual bundled source for the same code path: **4.10.38** uses a feature-detected `toHexUtil()` wrapper for the identical computation (`if (Uint8Array.prototype.toHex) return arr.toHex(); return Array.from(...).join("")`) — confirmed clean by reading that exact file, not by assuming an older version is automatically safer. Re-verified the full API surface this integration depends on (`getDocument`, `GlobalWorkerOptions`, `getViewport`, `.render()`, the worker's build path) was unchanged between 4.10.38 and 6.1.200 before committing to the downgrade. Pinned `pdfjs-dist` to `4.10.38`, removed the now-unnecessary `?v=1` (a version bump naturally produces a new content hash on its own).

**Confirmed working by the user**: PDF now fills the entire card correctly.

**Standing lesson:** when a maintainer (even a knowledgeable one, even in the exact right GitHub issue) suggests a specific fix, verify it against the actual installed source before trusting it — "legacy build" was a reasonable-sounding, plausible answer that turned out to be wrong for this specific bug once actually checked. The habit of reading real bundled source/type-definitions instead of trusting a description (already used for the pdf.js API itself in §4c) is what caught this before wasting a deploy cycle on it.

### Full list of root causes this one feature request needed, in the order they were found

1. Chrome's native iframe-based PDF viewer doesn't reliably resize with blob URLs (led to abandoning it for pdf.js)
2. Flex `align-items: center` doesn't stretch replaced elements to a percentage height
3. Replaced elements don't auto-size from `inset: 0` alone the way a plain `div` would
4. Apache had no `.mjs` MIME mapping, serving the pdf.js worker as `text/plain`
5. A content-hash-named asset doesn't bust caches when only server config (not the file) changes
6. pdf.js 5.x/6.x calls a browser-native method with no fallback, unsupported on the user's Chrome version

None of these were guessed and shipped on faith — each was confirmed with direct evidence (curl headers, `getComputedStyle`, real console/network output from the user's own browser, or the library's actual source) before being called "the fix," and after two fixes early on didn't resolve the symptom, that was correctly read as "wrong diagnosis," not "needs another attempt at the same theory."

**Standing lesson (spans §7–8):** two more real features this session were built by first checking whether the underlying data actually supported the naive version of the request (does `pending_imports` even return a count today? does `kit_vial_count` actually live at the product level or the price-row level?) rather than assuming the obvious UI addition was all that was needed — in both cases the honest answer changed the shape of what got built, and asking directly once that shape was ambiguous (rather than guessing) is what let a genuine user reconsideration (the Inventory-page pivot) get absorbed cleanly instead of costing a rebuild.
