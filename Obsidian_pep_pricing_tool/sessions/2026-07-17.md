# Session 2026-07-17

Fixed a bug where merging duplicate products, specs, or vendors in the Review Queue dedup
tooling silently orphaned `pc_price_history` rows under deleted ids, then backfilled every
historical merge back to the spec-merge feature's launch.

## pc_price_history orphaned on product/spec/vendor merges (root cause fix + backfill)

`pc_price_history` deliberately has no FKs â€” it's designed to survive `pc_prices` cascade-deletes
(see [[wiki/analyses/2026-07-03-price-history-spec]]) so a vendor/product purge doesn't wipe the
ledger. But that same no-FK design meant nothing caught it when a merge silently orphaned history
rows instead: `backend/api/products/merge.php`, `backend/api/products/spec_merge.php`, and
`backend/api/vendors/merge.php` all re-home `pc_prices` (and cart/stack items, aliases) onto the
merge winner before deleting the loser row, but none of them touched `pc_price_history` â€” its rows
were left pointing at product/spec/vendor ids that no longer existed. Not deleted (no FK to
cascade), just permanently unreachable, since `comparison/price_history.php`'s popover query only
ever looks up the *live* (vendor_id, product_id, specification_id, tier_kit_size) tuple. Confirmed
concretely before touching anything: vendor 11 / product 372 / spec 1282 had a live price row but
zero `pc_price_history` rows â€” the real history sat under dead `product_id` 449, orphaned by
today's `merge_product` (winner 372, loser 449, audit_id 4457, 14:37:02).

**Code fix** â€” added the missing repoint in the same transaction, before each script's loser
DELETE, mirroring exactly what each script already does for `pc_prices`:
- `products/merge.php`: two spots (matched-spec branch repoints both `product_id` and
  `specification_id`; no-equivalent-spec branch repoints just `product_id` â€” matches the existing
  `pc_prices` statements in each branch exactly).
- `spec_merge.php`: `specification_id` only, same WHERE shape as its `pc_prices` statement.
- `vendors/merge.php`: `vendor_id` only, dropped in right after the existing `pc_prices` line.

No FK added to `pc_price_history` â€” that's load-bearing (see the spec doc), the fix is purely
"repoint before delete," matching the pattern that already existed for carts/aliases in the same
scripts.

**Backfill** â€” `diagnostic_scripts/backfill_price_history_merge_orphans.php`: reads every
`merge_product` / `merge_specification` / `merge_vendor` action ever logged in
`pc_admin_audit_log` (306 total, oldest from 2026-07-03 â€” the day spec-merge shipped), and for
each one repoints any `pc_price_history` rows still sitting on that event's `loser_id` onto the
recorded `winner_id`. Chases transitively: if a winner was itself later merged away as a loser in
a subsequent event, it follows the chain to the final live id before repointing (one real case
hit this: audit_id 1568 chased loser 152 through intermediate winner 157 to final winner 2).
Idempotent by construction â€” once a row is repointed to the winner, the old loser id no longer
matches on a second pass. Dry-run (`--dry-run`) and the real run produced identical numbers:
**672 rows repointed across 172 merge events**. 4 events couldn't be resolved (audit_ids 14, 1572,
2896, 2897) â€” in each, the winner chain terminates at an id that doesn't exist in the target table
at all (never touched again by any later logged merge), a separate pre-existing data gap this
script correctly refuses to guess at rather than repoint to a wrong id.

**Verification**: vendor 11 / product 372 / spec 1282 now returns its real orphaned row (id 5981,
$197.00, changed_at 2026-07-17 14:35:13) on the exact popover-shaped query. Re-ran the backfill a
second time post-fix: 0 rows touched, same 4 unresolved â€” confirms idempotency. Verified the code
fix works going forward without touching production data: a throwaway product + winner/loser spec
pair + a price row + a history row were inserted inside an uncommitted transaction, spec_merge.php's
exact SQL sequence was run against them, the history row was confirmed repointed to the winner spec,
then the whole transaction was rolled back â€” verified via direct SELECT that no test rows persisted.

Deployed via `deploy.sh` (default code build+sync mode â€” no schema change needed). `php -l` clean
over SSH on all 3 touched merge scripts plus the new backfill script. Smoke check passed
(homepage/db/memcached/email all green); a ClamAV `clamd@scan` restart warning during deploy is
pre-existing infra noise, unrelated to any file this session touched.

Script archived at `diagnostic_scripts/backfill_price_history_merge_orphans.php` per this
project's one-off-script convention (no `migration_scripts/` directory exists here).
