## Per-suggestion Claude cost tracking (backlog #69)

**Schema** — `database/migrations/038_vendor_suggestion_claude_cost.sql`: adds
`pc_claude_call_log.vendor_suggestion_id INT UNSIGNED NULL` (positioned right after
`vendor_file_id`), an FK to `pc_vendor_suggestions(id) ON DELETE SET NULL` (mirrors `vendor_file_id`'s
own FK shape on this table exactly, confirmed via `SHOW CREATE TABLE` first), and its own index.
`database/schema.sql` mirrors the column/index inline in the `pc_claude_call_log` CREATE TABLE, but
the FK constraint had to move to a standalone `ALTER TABLE` placed after `pc_vendor_suggestions`'s own
CREATE TABLE — that table is defined earlier in the file (line ~325 vs ~420), before
`pc_vendor_suggestions` exists. **This broke `deploy.sh --all` on the first attempt** (`ERROR 1072
... Key column 'vendor_suggestion_id' doesn't exist` — the ALTER ran against the live DB's
already-existing `pc_claude_call_log`, where `CREATE TABLE IF NOT EXISTS` had been a silent no-op,
so the ALTER was the only place the column would have landed, and MariaDB has no `ADD COLUMN IF NOT
EXISTS`/`ADD CONSTRAINT IF NOT EXISTS` to make it safe to re-run). Pulled the ALTER back out of
schema.sql entirely — the real migration file is idempotent via `pc_migrations` tracking in
`migrate.sh` and is the actual source of truth for any existing DB; schema.sql now only carries the
column + index for a from-scratch install, with a `ponytail:` comment explaining the gap and where to
look if it matters later.

**Call chain** — added `?int $vendorSuggestionId = null` as the new trailing parameter (after
`vendorFileId`) on all three functions in `backend/lib/claude.php`:
`callClaudeExtraction(string $systemPrompt, array $userContent, string $model = CLAUDE_MODEL_DEFAULT, ?int $vendorFileId = null, ?int $vendorSuggestionId = null)`,
`callClaudeMessages(..., ?int $vendorFileId = null, ?int $vendorSuggestionId = null)`, and
`logClaudeCall(?int $vendorFileId, ?int $vendorSuggestionId, ...)` (this one's param sits right after
`vendorFileId` since it has no defaults at all — every param is positional/required there). Included
in the `INSERT INTO pc_claude_call_log` column list. Every caller checked by grep:
`vendor_file_processor.php:33` passes `(int)$file['id'], null`; `vendor_suggestions.php:188`
(`processSuggestion()`) passes `null, (int)$s['id']`; `claude.php:253`
(`callClaudeVendorContactParse()`) passes neither — both new params default to null there, correctly,
since that fallback path is never tied to a file or a suggestion.

**Admin endpoint** — `backend/api/admin/vendor_suggestions.php`'s GET handler runs one extra query
after the main suggestions list: pulls every `pc_claude_call_log` row matching any of the listed
suggestion ids (`WHERE vendor_suggestion_id IN (...)`), then sums `estimateClaudeCallCostUsd()`
per-row in PHP rather than summing raw tokens first — deliberate, since a reprocessed suggestion could
have call-log rows against two different models with different rates, and cost isn't summable from
pre-summed tokens across models. `estimated_cost_usd` added to each row alongside the other computed
fields, `null` when zero call-log rows matched (template-CSV suggestions, which never call Claude at
all).

**Frontend** — `VendorSuggestionsTab.vue` gets one line (`text-sm mono`, `$X.XXXX`) between the score
block and the extracted-prices table, `v-if="s.estimated_cost_usd != null"` — no dash placeholder,
since "not applicable" and "missing" need to look different here.
