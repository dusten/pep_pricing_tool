## 1. Security Fixes (before any feature work)

- **Referral codes**: were an 8-char truncated hex string (`substr(bin2hex(random_bytes(4)),0,8)`, 32 bits of entropy). Changed to a full 24-char random hash via the existing `generateToken(12)` helper. `pc_users.referral_code` and `pc_waitlist.referral_code` widened `VARCHAR(20)` → `CHAR(24)`.
- **Password hashing**: reviewed `PASSWORD_DEFAULT` (bcrypt) — already correct, auto-generates a random salt per hash. No change; flagged as a non-issue.
- **Admin/Expert relationship**: clarified all admins are Expert-and-above, but not all Expert users are admins. `requireTier()` in `backend/helpers.php` now returns immediately for `is_admin=true`, bypassing the tier check entirely — admin bypasses gates rather than the `tier` column being force-set to `expert`.

## 2. Mobile-First Nav Shell

- `TopBar.vue`: rebuilt with icon links — Admin (if `auth.isAdmin`), Settings, Logout — visible at every breakpoint. Dropped the old inline `ThemeToggle` cycling button (superseded by the Settings page).
- `Sidebar.vue` (desktop, hidden ≤768px): Dashboard, Calendar, Comparison, Pricing, Account. Removed the old Admin section + Logout button (now live in TopBar).
- New `BottomNav.vue` (mobile only, ≤768px): Dashboard, Calendar, Comparison — fixed to viewport bottom.

## 3. Comparison Table (`/comparison`) — real, replacing the "coming in Phase 3" placeholder

## 4. Calendar / Price History (`/calendar`) — new feature, not in original Phase 1 scope

# Session 2 — Nav unification, user settings, admin panel gaps, security/caching hardening

Second pass on the same day. User handed over admin-panel and user-settings requirements copied from two other apps in the same family (a personal-tracker app and a group-buy platform), explicitly excluding the XP/leveling system from either. Went through several rounds of scope clarification before building: self-service account deletion (not admin-mediated), one big pass rather than phased, ~1-2s p95 comparison query budget, and query replay as "log + admin-triggered re-run" rather than full request/response snapshots or log-only.

## 1. Nav — matches original ask literally this time

`Sidebar.vue` deleted outright. `AppLayout.vue` now renders `TopBar` + `BottomNav` at every breakpoint — desktop gets the same top+bottom nav shell as mobile, not a sidebar. `BottomNav.vue` gained the two items it was missing (Pricing, Account) to match what the old sidebar had. `PricingView.vue` — which had **zero nav** because it was never wrapped in `AppLayout` (public route, no auth required) — is now wrapped, so guests browsing pricing see the same shell as logged-in users.

## 2. User Settings (`/settings`) — went from one control to a real settings page

Rebuilt as stacked cards (not tabs — deliberate, matches the group-buy app's flattened-settings pattern since the section count is small enough to just scroll): Appearance (dark mode, kept at top — most-used toggle), Locale (timezone via `Intl.supportedValuesOf('timeZone')`, no library), Notifications (push toggle + PWA install card gated on `beforeinstallprompt`), Security (change password / change email / login history), Data export (one-click JSON download), Feedback (category pills, not a dropdown), Account deletion (password-confirmed, self-service).

Backend, all new under `backend/api/me/`: `password.php` (deletes every other `pc_sessions` row except the current one — simpler than the session-versioning column approach first attempted, see below), `email.php` + `auth/verify_email_change.php` (confirm-new-address flow, old address stays live until confirmed), `login_history.php` (new `pc_login_history` table, since `pc_sessions` rows get deleted on logout and can't serve as an audit trail), `export.php`, `sessions_revoke_all.php`. `me.php` gained a `DELETE` handler — cascades almost entirely for free via existing FK `ON DELETE CASCADE`/`SET NULL` constraints, no new cascade logic needed.

**Reverted mid-build:** initially added a `session_version` column to both `pc_users` and `pc_sessions` to support "change password without logging out the current session." Then noticed `auth/reset_password.php` already solves the identical problem with a plain `DELETE FROM pc_sessions WHERE user_id = ?` inside a transaction — reused that pattern (`DELETE ... AND id != ?` to keep the current session) instead, and backed out the schema column entirely. Worth remembering: check how the codebase already solves an adjacent problem before adding a new mechanism next to it.

Fixed a real bug found in `AccountView.vue`: "Sign out everywhere" was wired to `auth.logout()`, which only clears the *current* session — the label was a lie. Now calls the new `/api/me/sessions/revoke-all`.

## 3. Admin panel — filled every gap from the two reference specs

- **New Overview tab** — user/subscription/referral aggregates + `pc_admin_audit_log` feed (`backend/api/admin/overview.php`).
- **New System/Ops tab** — cache hit rate/memory (Memcached `getStats()`), DB connections/query counts (`SHOW STATUS`), slow queries deduplicated via `performance_schema.events_statements_summary_by_digest` (native MariaDB aggregation — deliberately not a hand-rolled slow-query-log file parser), maintenance run history (new `pc_maintenance_runs` table, populated by the existing cron jobs plus a new weekly `OPTIMIZE TABLE` job). Manual-refresh by default, off/1m/3m/5m poll selector, interval explicitly cleared in `onUnmounted` — a named failure mode from the source spec ("leaked interval timer keeps polling after nav-away") that's easy to introduce and was tested against here.
- **Users tab** — verification badge (`email_verified_at` was already selected by the API, just never rendered), new `test_account` flag/toggle, inline referral-tree viewer (`pc_referrals` already existed from the earlier session, just had no UI).
- **Waitlist tab** — bulk delete, CSV export (all or just-selected), status badges (pending/invited/confirmed derived from existing columns, no schema change).
- **Performance tab** — time-range selector (24h/7d/30d, defaults to 7d so it's never empty on open), device-type + path filters, top-pages/device-split breakdowns, raw recent-requests table. Required adding `device_type` to `pc_perf_metrics`, parsed server-side from `User-Agent` in `perf.php` (no client-side sniffing).

## 4. Comparison query logging + replay

New `pc_comparison_log` table. `comparison/index.php` now times itself (`microtime(true)` around the query) and logs `user_id`, the filter params, `duration_ms`, `result_count`, and a `slow_flag` (>1500ms). The actual row-building SQL got extracted from `comparison/index.php` into `backend/lib/comparison_query.php` as `runComparisonQuery()` specifically so the admin "re-run" endpoint (`admin/query_log_rerun.php`) calls the exact same code path instead of a hand-copied duplicate that could drift. Re-run does not write a new log row — it's a live admin debug action, not a real user query, and isn't cached.

## 5. Security — two rounds of spec, one direct contradiction the user resolved explicitly

User pasted security specs from both reference apps across several messages; the two disagreed on whether the Memcached rate limiter should fail open or fail closed when the cache backend is unreachable. Asked directly rather than guessing — **user chose fail closed** (reject with 503 rather than let brute-force through unlimited during a cache outage). `rateLimit()` in `helpers.php` rewritten to use atomic `add()`+`increment()` (closes a check-then-act race the old `get()`+`set()` version had) and fails closed.

Also landed: global exception handler (`set_exception_handler` in `public/index.php`, generic 500 to the client, real detail to `error_log`), real CORS allowlist active in all environments (previously dev-only, prod had zero CORS headers at all which happened to work but wasn't a deliberate allowlist), CSRF via Origin-header allowlist with a small **named** exemption list (currently just `perf` — the page-load timing beacon, which legitimately fires without a browser Origin in some cases) rather than a blanket "no Origin = allow," `requireAuth()` now re-checks `email_verified_at` on every request not just at login, transactional multi-step writes in `register.php` (user insert + referral edge + waitlist consumption all in one transaction), a self-referral guard (registrant IP checked against the referrer's `pc_login_history` — if it matches, the referral edge is silently not recorded and logged for review), enumeration-safe register (existing-but-unverified email resends the verification email, throttled to 1/hour, instead of leaking a distinct "already exists" response), timing-parity dummy `password_verify()` call on `forgot_password.php`'s not-found path, bcrypt-DoS password length caps (200 char max before hashing, on register/reset/change-password). CSP + HSTS + `LimitRequestBody` (32M, matched to `setup.sh`'s existing `post_max_size`) added at the Apache vhost level in both `price.conf` and `price-standalone.conf`.

Audited every mutating admin endpoint for `logAdminAction()` coverage — all of them already had it, no gaps found.

## 6. Caching — version-counter invalidation

New `cacheGet()`/`cacheBust()`/`cacheGroupKey()` helpers in `helpers.php`. Pattern: a write bumps one small version counter for a "group" (e.g. `admin_users`), which makes every existing Memcached entry under that group unreachable without having to enumerate or delete them individually — old entries just expire on their own TTL. Applied to `admin/users.php` (30s TTL, shared version across all tier-filter variants), `admin/waitlist.php` (30s), public `app_settings.php` GET (5min — slower-changing reference data; admin's own GET stays uncached so an admin sees their own edit immediately). Deliberately never applied to `comparison/index.php` or `/api/me` — those must always reflect the instant they were written. Caching fails open by design (a cache miss just means "compute fresh"), unlike the rate limiter which fails closed — different risk profile, same helper file, worth not confusing the two.

## 7. deploy.sh — split into 4 explicit modes, then a smoke check added

Was 2 modes (`--sync-schema`, and everything else = full deploy). Now 4, each its own function: `--sync-files` (rsync + remote reload steps, no npm build — for backend-only changes), `--build` (same as the old default: npm build then sync), `--sync-schema` (unchanged), `--all` (schema sync then build+sync). Default with no flag is still `--build` for backward compatibility with existing muscle memory / any external tooling calling `bash deploy.sh` bare.

After a manual `curl` smoke test post-deploy, user asked why that isn't automatic — added a `smoke_check()` function (curl homepage + `/api/app-settings`, both must 200, non-zero exit otherwise) that now runs automatically at the end of every `sync_files()` call, so it fires after `--sync-files`, `--build`, and `--all`. Left the real fix — a proper `/api/health` endpoint exercising DB/Memcached/email — as `CLAUDE.md` backlog item 3, which already existed and already covers this; didn't duplicate it.

## 8. Shipped to production

Full build merged to `main` (fast-forward, no conflicts) and pushed. `bash deploy.sh --all` run against `price.themightygroupbuy.com`: schema migration `003_settings_admin_expansion.sql` applied cleanly, frontend rebuilt, files synced, smoke check passed (homepage 200, `/api/app-settings` 200).

## 9. Post-deploy verification found a real bug — migration 003 silently never ran on prod

Asked to verify the deploy was actually working, not just that the smoke check passed. Curled every new/changed endpoint unauthenticated (expecting 401s) — `api/auth/verify-email-change?token=x` came back `500` instead of the expected `404`.

**Root cause:** `schema.sql`'s seed data pre-registers `INSERT IGNORE INTO pc_migrations (filename) VALUES ('003_settings_admin_expansion.sql')`. `deploy.sh --sync-schema` runs `mysql < schema.sql` *before* `migrate.sh`. So schema.sql marked migration 003 "already applied" a few seconds before `migrate.sh` ever got a chance to run its real `ALTER TABLE ADD COLUMN` statements — `migrate.sh` saw the row, printed "already applied," and skipped it. `schema.sql`'s own `CREATE TABLE IF NOT EXISTS pc_users` is a no-op against prod's already-existing table, so the new columns (`timezone`, `push_enabled`, `test_account`, `pending_email`, `email_change_token`, `pc_perf_metrics.device_type`) never actually landed — while `pc_migrations` confidently claimed otherwise. Same bug pattern applied to migration 002's seed row too, but that one happened to be harmless since 002 had genuinely already been applied for real in Session 1, before this bug was introduced.

**Why it existed:** copied the existing "seed `001_initial.sql` as applied" convention without noticing the difference — 001 is safe to pre-seed because `schema.sql`'s `CREATE TABLE` *is* 001's end state on a fresh install. Migrations 002/003 are `ALTER TABLE` deltas against an *existing* install; pre-seeding them only makes sense for fresh installs, and actively breaks existing ones.

**Fix (commit `ed85e80`):**
- Migration 003 rewritten to use `ADD COLUMN IF NOT EXISTS` (MariaDB 10.0.2+) on every column — now idempotent regardless of whether it's pre-seeded or not, so it's safe on both a fresh install (where `schema.sql`'s `CREATE TABLE` already has the columns) and an existing one (where it doesn't yet).
- Removed the premature `INSERT IGNORE` seed rows for 002 and 003 from `schema.sql` — only `001_initial.sql` stays seeded, matching the original (correct) convention. Left a comment in `schema.sql` explaining why, so a future migration doesn't repeat this.
- On prod: deleted the false `pc_migrations` row for 003 via SSH, then re-ran `bash deploy.sh --sync-schema` — migration applied for real this time (`1 applied, 1 skipped`), confirmed via `SHOW COLUMNS` over SSH and the endpoint re-check now returning the expected `404`.

**Standing rule going forward:** never add a migration filename to `schema.sql`'s `pc_migrations` seed data beyond `001_initial.sql`. If a migration needs to be safe on both fresh and existing installs, make the migration statement itself idempotent (`IF NOT EXISTS` / `IF EXISTS` where the DB engine supports it) rather than gaming the tracking table.

Also did a live browser pass post-fix (via claude-in-chrome, DOM/JS checks since screenshot capture was timing out): Pricing page nav confirmed (5 BottomNav items + TopBar), all 12 admin tabs render, System tab pulls live cache/DB stats with no console errors, Users tab shows the new Verified/Test columns, Settings page shows all 7 new cards with the timezone dropdown populated (418 IANA zones via `Intl.supportedValuesOf`).

Profile (display name/email), Subscription, and a redesigned Refer-a-Friend card (icon header, code+copy, Joined/Converted/Credit-Earned stat tiles backed by a new `/api/me/referral-stats` endpoint) all moved into `SettingsView.vue`. `BottomNav` dropped its "Account" item — Settings is now the sole entry point, consistent with `TopBar` which never had a separate Account icon.

- **Pricing page buttons weren't aligned horizontally** — `.pricing-card` was `display: block`, so a button's vertical position depended on how much text sat above it (Expert's longer feature list pushed its button ~40px lower than the others). - **Guest TopBar showed "Settings"/"Sign out"** even when logged out — neither made sense (Settings would just bounce to login, Sign out had nothing to sign out of). ## 2. Caching audit (user-requested: "let's do a full review of where we can use caching")

Found `getAppSetting()` — checked on nearly every request via the unconditional maintenance-mode gate in `public/index.php` — had only a per-*request* static-array cache, hitting the DB fresh on every single request, site-wide. That was the highest-value fix. Backed it with the same Memcached cache as `app_settings.php`'s public GET.

Expanded caching from 3 endpoints to 10 across 18 files:
- `getAppSetting()` (see above)
- Comparison results + filters + calendar share a `pricing_data` cache group, since the underlying data (prices/vendors/products) is identical for every user regardless of who's asking — only the free-tier quota check is personal, and that logic was kept completely separate from the now-cached row-fetch. Busted from every real write path: file processing (the Claude-extraction price importer), vendor CRUD, product CRUD/merge/aliases.
- Admin lists (`vendors`, `products`, `feedback`) — same version-counter pattern as the existing `users`/`waitlist` caches.
- Admin dashboards (`overview`, `performance`) — TTL-only (30s), no bust wiring; too many scattered write paths feed these to make precise invalidation worth it for a dashboard tile.

**Then went further** (asked directly: "confirm maintenance/waitlist/session are all in Memcache"): maintenance mode and waitlist mode were both already covered by the `getAppSetting()` fix, but **session/token validation** — `requireAuth()`'s `pc_users`/`pc_sessions` JOIN, run on literally every authenticated request — was not cached at all, easily the highest-frequency query in the app. Added a 60s-TTL cache keyed by token hash, with precise bust-on-write wired into logout, password change, email change, and account deletion. Caught and fixed a subtlety before shipping: caching the *entire* `requireAuth()` result (not just token validity) meant a self-edit (theme/name/timezone) could appear not to have saved for up to 60s on the next `fetchMe()` — violates the "a save should never wait out a TTL" rule from earlier in the day. ## 10. Claude key went live — real extraction pipeline debugged end-to-end, then extended with caching, batching, and image support

User added `ANTHROPIC_API_KEY` to prod and uploaded a real vendor XLSX (`Purelypep`/vendor id 3, ~265 rows, RMB pricing). First real run through the extraction pipeline surfaced two independent, stacked bugs:

1. **`xlsx_reader.php` silently dropped every text cell.** `sharedStrings.xml` declares a default `xmlns` on `<sst>`; SimpleXML's plain property access (`$doc->si`) crosses that namespace fine, but `SimpleXMLElement::xpath('.//t')` does not — every shared string resolved to `''` while numeric cells (which don't use shared strings) kept working, so the bug looked like "half the columns are blank" rather than a total parse failure. Confirmed empirically via SSH (`count($doc->si) == 265` but `xpath(".//t")` returned 0 on all of them) before touching code. Fixed by using `strip_tags($si->asXML())` instead of xpath — sidesteps namespaces entirely and also naturally concatenates multi-run rich text.
2. **`claude.php` was silently burning its own token budget.** Two compounding issues in `callClaudeMessages()`: (a) no `thinking` param was set, and on Sonnet 5 — unlike Opus 4.7/4.8 — *omitting* `thinking` doesn't mean "off," it silently runs adaptive thinking, which ate 5,516 of the 8,000 `max_tokens` before the real JSON output even started; (b) response parsing read `content[0]['text']` unconditionally, so once a thinking block occupied index 0, extraction broke outright. Fixed by explicitly setting `thinking: {type: "disabled"}` (this is deterministic extraction, not reasoning — no thinking needed), raising `max_tokens` 8000→16000, and scanning `content[]` for the first `type === 'text'` block instead of trusting index 0.

Verified the fix against the real file server-side (`sudo -u apache php -r` calling the actual updated functions) before deploying: 95 real price rows extracted correctly, RMB→USD conversion confirmed accurate (¥95 → $13.19 at the 7.2 rate), TB-500/variant warnings fired correctly on ambiguous combo products. Deployed, then closed the loop through the real `/api/files/1/process` endpoint (not just the isolated test): 5 rows auto-committed (exact catalog matches), 88 correctly held in `pc_pending_imports` for review since none of those products existed yet — confirms the hard-review-queue design is working as intended, not silently auto-creating catalog entries.

**Follow-on work, same session, three separate asks:**

- **Prompt caching.** User asked whether caching made sense since every file processed shares the same system prompt. Built `POST /api/files/process-all` (`backend/api/files/process_all.php`) — loops every `pending` file sequentially through the same `processVendorFile()` used by the single-file endpoint (no logic duplication), routes oversized PDFs to the existing async queue same as before, returns per-file results. Not yet tested against a real upload (the example image arrived pasted inline in chat, no accessible file path to script against) — worth confirming with a real upload through the UI.

**Timeout bug found while building the batch feature, fixed at all three layers it lived in:**

Batch processing meant one HTTP request could now run many sequential Claude calls — needed PHP and Apache to both tolerate that. Investigation found the *single*-file endpoint was already quietly broken the same way and had just never been noticed (small/fast files never triggered it):

- **PHP:** `/etc/php.d/99-tmgb.ini` sets `max_execution_time = 30`, well under `claude.php`'s own 180s curl timeout — meaning a slow single-file extraction could already have been getting killed by PHP itself before curl's timeout ever mattered. Fixed with `set_time_limit(200)` in `process.php` and `set_time_limit(900)` in the new `process_all.php` (confirmed this ini setting is a plain overridable directive, not an unoverridable FPM `php_admin_value`, before relying on `set_time_limit()`).
- **Apache — two wrong turns before the right one.** First attempt scoped `Timeout 200`/`900` inside `<LocationMatch>`/`<Location>` blocks for just the two file-processing paths — `httpd -t` rejected it: `Timeout` is server-config/vhost scope only in Apache, not valid in directory-level contexts at all. Second attempt swapped to `ProxyTimeout` (the mod_proxy-specific directive, expected to support directory scope) — rejected the same way on this Apache build. Gave up on per-path scoping and landed on a vhost-level `Timeout 900` override instead, in the `<VirtualHost *:443>` block directly — broader (all price-app requests, not just these two endpoints) but the only form Apache actually accepts, and it still only affects the price vhost, not the shared `grp` vhost in `00-security.conf`. Both failed attempts were caught safely by running `httpd -t` before every `systemctl reload` — confirmed zero downtime both times (site stayed `200`/`200` throughout) before the working version shipped.

**Standing lesson:** `deploy.sh` doesn't touch `/etc/httpd` — vhost config changes need a manual `scp` + `httpd -t` + `systemctl reload` outside the normal deploy flow. The permission system correctly treats each push of an *edited* config as needing its own fresh "yes, go ahead," even seconds after a prior approval of a different (broken) version of the same file — approval doesn't generalize across edits. Also: `httpd -t` before every `systemctl reload` is what made it safe to iterate live on Apache config through two wrong guesses without any real risk — worth doing by habit whenever touching `/etc/httpd` on this box going forward.

## 11. A real local vendor file (`Peptide0616.xlsx`) exposed a wrong-sheet bug, a duplicate-product crash, and a tier-clamping bug — all three found by actually working the file through end to end

User pointed at a real file outside the repo (`/home/dusten/projects/pep_cal/pepper_vendor/Peptide0616.xlsx`, part of a personal folder of ~20 real vendor price lists) asking why extracted values didn't match what they saw when opening it. Not a formula/cached-value issue — the workbook has two sheets, and `xl/worksheets/sheet1.xml` (which `xlsxToText()` hardcoded reading) is a **hidden** tab (`state="hidden"` in `workbook.xml`) holding flat internal per-box RMB pricing; the real, visible tab ("Sheet2", the workbook's own `activeTab`) holds the actual tiered USD quote (`≥1kit / ≥10kits / ≥50kits` columns, 149 rows). Physical `sheetN.xml` filenames have no reliable relationship to tab order/visibility — confirmed by cross-referencing `workbook.xml`'s `<sheets>` list against `workbook.xml.rels`' `rId → target` mapping.

Fixed by adding `xlsxPickSheetFile()`: prefer the workbook's own `activeTab`, fall back to the first non-hidden sheet if that's hidden, fall back to sheet1 if everything's hidden. Also made it surface a note (`"This file has N sheets (M visible) — only 'X' was read..."`) into `processing_notes` — same channel Claude's own extraction warnings already use — so a future multi-sheet file gets flagged for a human to double-check instead of silently trusting whichever sheet got picked. This is the direct answer to "how does this get managed in the future": visibility at review time, not a promise that sheet-picking heuristics are always right.

Turned out this exact file (same filename, same 26,200-byte size) was already live in production as vendor 3/file 1 — meaning the "95 rows successfully extracted" result verified earlier in §10 had, the whole time, been extracting the wrong hidden sheet. Real cleanup required three full delete-and-reprocess cycles because each reprocess attempt surfaced a *new*, previously-latent bug once real data actually flowed through the full pipeline:

1. **Duplicate-product crash blocked the whole review queue.** User: "I couldn't accept any more products as I was stuck on the NAD+ 500mg." Root cause: `pc_products.canonical_name` is `UNIQUE`, but `pending_imports.php`'s approve handler calls `createProduct()` whenever `candidate_product_id` is null — and that field is computed once, at file-processing time, before any review-session approvals exist. NAD+ has three spec rows (100mg/500mg/1000mg), all flagged `new_product` since NAD+ didn't exist in the catalog yet. Approving the 100mg row created "NAD+"; approving 500mg then tried to create "NAD+" *again* and hit the UNIQUE constraint. The backend correctly caught and returned a 500 — but `ReviewQueueTab.vue`'s `approveImport()`/`rejectImport()`/`approveCoa()`/`rejectCoa()` had zero error handling, so the promise rejection vanished silently and the same card just sat there forever with no message. Fixed both: backend now re-checks `findExactProductMatch()` immediately before `createProduct()` (the stale candidate isn't trustworthy, a fresh check is), frontend now wraps all four actions in try/catch + `alert(err.message)`, matching the existing pattern already used in `FilesTab.vue`. This would have hit on *any* multi-spec new product, not just NAD+ — a systemic bug, not a one-off.
2. **Claude added a preamble sentence on the large reprocess**, despite the system prompt's explicit "no preamble" instruction — never observed on the smaller 95-row file, only on this 149-row one. `json_decode()` choked on the leading prose. Fixed by extracting the outermost `{...}` from the response text before parsing, rather than trusting the whole body is pure JSON — robust regardless of why a preamble shows up, not a prompt-wording patch that the model could just as easily ignore a different way next time.
3. **max_tokens truncation.** 149 rows × up to 3 tiers ≈ 450 output objects — confirmed via `stop_reason: "max_tokens"` and text cut off mid-object at the 16,000-token ceiling tuned earlier against a smaller, non-tiered file. Raised to 48,000 (~20% headroom over the observed ~88 tokens/object density), curl timeout 180s→400s, and `process.php`'s `set_time_limit()` 200→450 to match.
4. **Tier-clamping data corruption, caught by inspecting the actual committed rows rather than trusting row counts.** `vendor_file_processor.php` and `pending_imports.php` both clamped any `tier_kit_size` outside `{1, 10, 100}` down to `1` — hardcoded against the *assumption* that those are the only tiers vendors use. This vendor's real tiers are 1/10/**50**. Confirmed via the raw `pending_imports.raw_json` that Claude correctly extracted `tier_kit_size: 50` — the corruption was entirely on the commit side: clamping 50→1 collides with `pc_prices`' `UNIQUE(vendor_id, product_id, specification_id, tier_kit_size)` key, so the ≥50kits price either overwrote or got overwritten by the real tier-1 price, silently, with no error and no trace once committed. The DB column itself (`TINYINT UNSIGNED`) was never the constraint — it was three PHP call sites (two commit-time clamps, one prompt sentence) all independently hardcoded to the same wrong assumption. Fixed all three: clamps now just floor/ceiling to the column's actual range (`min(255, max(1, ...))`), prompt no longer says "1, 10, or 100," says breakpoints vary by vendor.

Final state after the third clean reprocess: 21 rows committed, 399 in the review queue, BPC-157 10mg correctly showing all three tiers ($46 / $38 / $29, sensible bulk-discount curve). Each of the three bugs was caught by actually inspecting real committed data (row counts, raw JSON, specific product/spec/tier combinations) rather than trusting a "processing complete" status alone — a `processing_status = 'complete'` row told us nothing about whether the *data* was actually correct.

**Two new backlog items filed as a direct result** (added to `CLAUDE.md` and `wiki/entities/phase-roadmap.md`, items 6–7): `pc_prices.tier_kit_size` is `TINYINT UNSIGNED` (max 255) — a genuine 1000-kit tier would now silently clamp to 255 instead of erroring, needs a column-widening migration; and `comparison_query.php` hardcodes `tier_kit_size = 1`, so even with every tier now stored correctly, only the 1-kit price is visible anywhere in the actual product — bulk pricing is fully invisible to admins and end users until that query (and whatever UI renders it) gets built out.

**Separately: prompt caching was confirmed not yet active, correctly diagnosed as expected rather than broken.** Console dashboard said "You're not using prompt caching" after processing 3 files back to back. Verified directly against the API (not the dashboard, which can lag) via two raw back-to-back calls: `cache_creation_input_tokens: 0` on both, including the *first* call — meaning no cache write was even being attempted, not a timing issue. Root cause: `buildExtractionSystemPrompt()`'s live product/alias list is tiny right now (only 4 products in the catalog at the time), making the whole system prompt ~470 tokens — under the minimum cacheable prefix length for any Claude model tier (roughly 1024–2048+ tokens). Nothing to fix; the `cache_control` marker is correctly formatted and will start actually engaging on its own as the catalog grows past that floor, no code changes needed. Explicitly did not recommend padding the prompt to force caching — would cost more than it saves at this catalog size.
