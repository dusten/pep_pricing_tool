## 12. The sorting bug — found, then fixed, same day

User asked to actually investigate before assuming anything (having said one message earlier "don't want you to change anything yet, just wanted to see gaps"). Checked the three places specs get sorted directly rather than trusting the blueprint's stated intent alone:

- `comparison_query.php` (public Comparison table) — `ORDER BY ... s.numeric_value ASC` — correct
- `products/show.php` (Products tab) — `ORDER BY numeric_value` — correct
- `vendors/show.php` (Inventory tab **and** the Vendors-tab price list — they share this one query) — `ORDER BY ... s.spec_label ...` — **wrong column**, sorting the display string ("10mg") instead of the numeric column sitting right next to it

User supplied a concrete screenshot (Tirzepatide, Inventory tab: 10, 15, 20, 30, 5, 60mg) that pinned the bug precisely — that's exactly correct alphabetical order (first-character comparison 1,1,2,3,5,6), confirming a lexicographic-vs-numeric mixup, not bad data. Root cause traced to the exact line, written earlier this same session when the vendor price-list feature was first built — self-inflicted this session, not a long-standing bug, and notably *not* present in either of the two queries that were already correct.

User then explicitly authorized the fix ("Yes, fix it") as a separate step from the investigation. One-word change (`spec_label` → `numeric_value`), deployed, and verified against live data post-deploy rather than just trusting the diff: a real vendor's Tirzepatide specs now return `5, 10, 15, 20, 30, 40, 50, 60, 80, 100mg` in that exact order. Full writeup in [[wiki/analyses/2026-07-03-blueprint-vs-actual]].

**Standing lesson:** the founding document explained *why* sorting was designed to be numeric from day one — it did not, by itself, explain why the user was *seeing* it broken, since the query that matters most (the public comparison table) was already correct. Checking the live queries directly, plus the user's own concrete screenshot, is what actually found the bug — re-reading the spec harder would not have.

**Process lesson worth keeping**: this whole arc ran as three explicitly separate, user-gated steps — review only ("just wanted to see gaps"), then investigate only ("go find it"), then fix ("yes, fix it") — never collapsed into one. Worth defaulting to that same staging on any core pricing/comparison-logic change going forward, not just when explicitly told to slow down.

## 13. New session, same day — Products tab: Edit-button row misalignment + spec merge

User opened a fresh conversation later the same day and reported two Products-tab (admin) bugs from a screenshot: a border/alignment glitch on the row below the Edit button, and no way to merge two duplicate spec rows for the same product (e.g. 5-Amino-1MQ had both a "5mg" and a "5mg\*10vials" row, both at 5.0000mg, needing to become one).

**Duplicate-spec merge** — built as a spec-level sibling to the existing product-merge feature: new `POST /api/products/specifications/{id}/merge` (`backend/api/products/spec_merge.php`, route registered in `public/index.php`) moves `pc_prices` rows from the loser spec onto the winner (`UPDATE IGNORE`, so any vendor/tier collision is dropped) then deletes the loser spec row — collided price rows are cleanly cascade-deleted via the existing `specification_id` FK (`ON DELETE CASCADE`), same as `merge.php`'s product-level merge already relies on. Frontend: added a second "Merge into…" dropdown per spec row in `ProductsTab.vue`, scoped to sibling specs of the same product. Verified live against the real 5-Amino-1MQ duplicate (5mg / 5mg\*10vials) — user confirmed correct.

**Row-alignment bug — real root cause found via DOM geometry, not guesswork.** First pass fixed a different, real bug in the same table (the spec-editing row's "Move to…" `<select>` had no explicit width, so the global `select { width:100% }` rule blew it up to ~930px inside the flex row, dragging the trailing vendor-count text out from under its column into the Edit-button's column) — but that wasn't what the user was pointing at. User's second screenshot showed the plain (non-editing) product list, so the specs-row fix was verified correct but off-target. Chrome screenshot capture was broken all session (`Page.captureScreenshot` timed out repeatedly, even after reload/navigate — a tool-level issue, not a page-frozen issue, confirmed because `javascript_exec` kept working fine throughout); diagnosed entirely via `getBoundingClientRect()` + `getComputedStyle()` pulled through `javascript_exec` instead. That surfaced the actual bug: `.actions` (the `<td>` holding the Edit/Save/Cancel buttons) had `display: flex` in its CSS rule, which overrides a table cell's default `display: table-cell` — so that one cell stopped stretching to the row's height and sized to its own button instead, leaving its bottom border ~11px above the rest of the row's border line. Fixed by dropping the flex and spacing the Save/Cancel pair with `.actions button + button { margin-left: 4px }` instead, confirmed by comparing all 6 cells' `top`/`bottom` rects post-deploy (now identical, `226.5 → 281.5` across the row).

**Standing lesson**: a user's second screenshot after "no, still wrong" can be pointing at a completely different location in the UI than the first fix addressed — check what state (editing vs. closed) the new screenshot actually shows before assuming it's the same spot, and confirm the true fix by comparing raw geometry (`getBoundingClientRect`) rather than trusting a visual re-guess. Also: `display: flex` on a table `<td>` is a recurring trap — it silently opts that one cell out of the table's row-height model.

## 14. New session, same day — pending-imports FK violation

User reported a bug with only the raw error, no other framing: `SQLSTATE[23000] ... pc_specifications_ibfk_1 ... FOREIGN KEY (product_id) REFERENCES pc_products`. Grepped every `pc_specifications` write path; the normal file-processing commit (`vendor_file_processor.php`) always resolves `product_id` fresh via `findExactProductMatch`/`createProduct`, so it's safe. The approve endpoint (`backend/api/vendors/pending_imports.php`) wasn't: it trusts a stored `candidate_product_id` (computed at file-upload time by fuzzy match) or an admin-supplied `product_id`, neither of which is re-validated — if that product was since deleted (e.g. `products/merge.php` deletes the merge-loser), the id is dead and the `pc_specifications` insert trips the FK.

Fix: before trusting `$productId`, confirm the row still exists in `pc_products`; if not, fall through to the existing `findExactProductMatch(...) ?? createProduct(...)` re-resolution path that was already there for the "stale null candidate" case. One file, `pending_imports.php`. Committed (`a1d8880`), fast-forward merged onto `main` (working in a worktree, with an unrelated uncommitted local WIP — the `tier_kit_size` TINYINT→SMALLINT widening — stashed and restored around the merge so it wasn't disturbed or accidentally committed), pushed, and deployed via `bash deploy.sh` (backend-only change, default build+sync mode, no schema touched). Live smoke check passed (homepage/db/memcached/email).

Deploy's remote post-deploy step logged `⚠ ClamAV signatures copied but clamd@scan restart failed` — expected, not a regression: `database.clamav.net` is IP-blocked from this EC2 range (see `deploy.sh:95-97`), and the user hasn't yet sourced a way to get fresh `.cvd`/`.cld` signature files onto the box, so clamd has nothing valid to start with. Doesn't affect the app.

**Process note**: user explicitly said "don't ask in the future" after this fix — for a bug reported as a concrete error (stack trace / exact SQL error / constraint name) rather than a vague symptom, the error itself is authorization to investigate-and-fix in one pass, no staged confirmation. Recorded in `[[feedback_stage_consequential_fixes|memory]]` as the counterpart to the sorting-bug precedent, which *does* still warrant staged gating — that one turned on a product-behavior judgment call, this one didn't.
