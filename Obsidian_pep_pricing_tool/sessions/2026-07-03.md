## 12. The sorting bug — found, then fixed, same day

User asked to actually investigate before assuming anything (having said one message earlier "don't want you to change anything yet, just wanted to see gaps"). Checked the three places specs get sorted directly rather than trusting the blueprint's stated intent alone:

- `comparison_query.php` (public Comparison table) — `ORDER BY ... s.numeric_value ASC` — correct
- `products/show.php` (Products tab) — `ORDER BY numeric_value` — correct
- `vendors/show.php` (Inventory tab **and** the Vendors-tab price list — they share this one query) — `ORDER BY ... s.spec_label ...` — **wrong column**, sorting the display string ("10mg") instead of the numeric column sitting right next to it

User supplied a concrete screenshot (Tirzepatide, Inventory tab: 10, 15, 20, 30, 5, 60mg) that pinned the bug precisely — that's exactly correct alphabetical order (first-character comparison 1,1,2,3,5,6), confirming a lexicographic-vs-numeric mixup, not bad data. Root cause traced to the exact line, written earlier this same session when the vendor price-list feature was first built — self-inflicted this session, not a long-standing bug, and notably *not* present in either of the two queries that were already correct.

User then explicitly authorized the fix ("Yes, fix it") as a separate step from the investigation. One-word change (`spec_label` → `numeric_value`), deployed, and verified against live data post-deploy rather than just trusting the diff: a real vendor's Tirzepatide specs now return `5, 10, 15, 20, 30, 40, 50, 60, 80, 100mg` in that exact order. Full writeup in [[wiki/analyses/2026-07-03-blueprint-vs-actual]].

**Standing lesson:** the founding document explained *why* sorting was designed to be numeric from day one — it did not, by itself, explain why the user was *seeing* it broken, since the query that matters most (the public comparison table) was already correct. Checking the live queries directly, plus the user's own concrete screenshot, is what actually found the bug — re-reading the spec harder would not have.

**Process lesson worth keeping**: this whole arc ran as three explicitly separate, user-gated steps — review only ("just wanted to see gaps"), then investigate only ("go find it"), then fix ("yes, fix it") — never collapsed into one. Worth defaulting to that same staging on any core pricing/comparison-logic change going forward, not just when explicitly told to slow down.
