# Session 2026-07-19

## Slow-query log processing methodology distilled into a portable prompt

`grp.themightygroupbuy.com` (the sibling project) shares the same physical MySQL server and
therefore the same `mysql.slow_log` table as this app — meaning the slow-query capture pattern built
here over several sessions (collector built 2026-07-01, first triage pass 2026-07-04 which found and
fixed the `findExactProductMatch()` `LOWER()` index-defeat bug, noise-filtering pass 2026-07-10, a
regression fix 2026-07-14 after `pc_import_slow_queries` events silently stopped firing) is directly
reusable there, but only existed so far as scattered wiki entries and commit history rather than one
citable reference.

Read back through the full lineage and distilled it into a single reusable build/triage/re-audit
prompt, filed as [[wiki/concepts/slow-query-log-capture-pattern]] — covers how to build the collector
if grp doesn't have one yet, how to triage a first pass (the noise-vs-signal split that mattered here:
90%+ of raw hits were often one query or a maintenance job, not real problems), and how to re-audit
periodically without re-deriving the method from scratch each time.

No code changes this session — pure knowledge-capture, read-only against this app's own history.
