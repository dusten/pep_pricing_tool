## Activity dashboard fix: real search logging, DAU, admin/test toggle, immediate cache bust

Built out the previously-diagnosed admin Activity dashboard fixes (search logging fully broken via
`pc_query_log`'s dedup/quota-only nature, "clicks not showing" as a stale-cache issue, DAU buildable
off existing `pc_sessions.last_seen_at`), plus the product-decided admin/test-activity toggle.

**Migration 039**: new `pc_search_log` table (`id`, `user_id`, `created_at`, FK to `pc_users`,
indexed on `created_at` and `(user_id, created_at)`) — an unconditional per-request log, mirrored
into `database/schema.sql`. Explicitly does not touch `pc_query_log` or its dedup/quota logic.

**`backend/api/comparison/index.php`**: one new unconditional insert into `pc_search_log` right
after filter parsing, before the existing `$isFree` quota block — every hit logs regardless of
tier/admin, matching "a search should be logged any time you are viewing the comparison page."

**`backend/api/admin/activity_trend.php`**: generalized `$METRICS` to carry `date_col`,
`distinct_col`, `user_col` per entry; `?include_internal=1` toggle (default excluded) building a
`JOIN pc_users u ON u.id = {table}.{user_col}` + `u.is_admin = 0 AND u.test_account = 0` exclusion
for every metric except `signups` (filters `pc_users` directly, no self-join); `searches` repointed
`pc_query_log` → `pc_search_log`; added `daily_active_users` (`pc_sessions`,
`COUNT(DISTINCT user_id)` on `last_seen_at`). Cache TTL 600s → 60s, cache key varies by the toggle
(`with_internal` / `external`).

**Cache-busting** (added mid-task per direct user follow-up, on top of the shortened TTL —
belt-and-suspenders): `cacheBust('admin_activity_trend')` added right after every write that feeds
a metric — `link_click.php`, `login.php`, `register.php`, the new search-log insert in
`comparison/index.php`, `export_full.php`, `comparison/export_csv.php`,
`comparison/export_xlsx.php`. Left DAU (`pc_sessions.last_seen_at`) on TTL-only — it updates on
every single authenticated request site-wide via `requireAuth()`, so busting there on every request
would be pointless; 60s staleness is fine for a daily metric.

**Frontend** (`OverviewTab.vue`): "Include admin/test activity" checkbox next to the Activity
section header (default unchecked), `loadTrend()` extracted so both `onMounted` and the checkbox's
`@change` call it with `?include_internal=1` appended when checked; `daily_active_users` added to
the `METRICS` array (renders with the existing generic bar-chart card, no new component).

**Bug found during verification** (not part of the original diagnosis): the exclusion JOIN made
`created_at` ambiguous for every metric table (all of them also have their own `created_at`),
producing a live 500 (`SQLSTATE[23000]: Column 'created_at' in field list is ambiguous`) on the
default excluded view. Fixed by table-qualifying `date_col`/`distinct_col` and the `pc_users` bare
columns; verified with a second deploy + retest.

## Referral rewards: dollar credit → free months (backlog #4 pre-Stripe correction)

This project's own wiki had flagged this exact change as needed before Stripe billing gets built
([[wiki/entities/phase-roadmap]] backlog #4, [[sessions/2026-07-03]]). Confirmed live first:
`account_credit_usd` and `pc_referral_credits` were 100% cosmetic — never written or consumed by any
code path. Built the real grant mechanism for the first time, in months instead of dollars.

**Migration 040**: `pc_referral_credits` — dropped `amount_usd`, added `months_granted SMALLINT
UNSIGNED NOT NULL DEFAULT 0`. `pc_app_settings.referral_credit_usd` ('5.00') deleted, replaced with
`referral_months_free` ('2', matching `annual_discount_months_free`'s existing default). Mirrored in
`schema.sql` (table def + seed data) for fresh installs.

**Grant logic** — `backend/api/admin/users_show.php` (PATCH `/admin/users/{id}`, the same endpoint
admins already use today, no new UI): fetches `tier_status`/`referred_by_id` before the update; after
applying it, if `tier_status` genuinely transitioned to `'active'` from something else and the user
was referred, checks `pc_referral_credits` for a dedupe (`granted_at IS NOT NULL`). If clear: upserts
the credit row, extends the referrer's `tier_renews_at` by `getAppSetting('referral_months_free', '2')`
months via `DATE_ADD(GREATEST(COALESCE(tier_renews_at, NOW()), NOW()), INTERVAL ? MONTH)`, bumps their
`tier_status` to `active` only if not already active/trialing (never touches `tier`), logs via
`logAdminAction('referral_reward_granted', ...)`. Wrapped in the codebase's standard
`beginTransaction`/try-commit/catch-rollback shape.

Two call sites not in the original task list were found via grep and fixed (would have thrown a live
SQL error on the dropped column): `backend/api/me/export.php` and `backend/api/admin/overview.php` +
its `OverviewTab.vue` dashboard tile (`total_credited_usd` → `total_months_credited`).

**Frontend**: `SettingsView.vue` "Refer a Friend" stat tile → "Months Earned" (no `$`), copy → "you
earn free months added to your subscription"; `UsersTab.vue` referred-user list → "credited N mo";
admin `SettingsTab.vue` field → "Referral reward (months)".

**Verification**: no real referred-and-unconverted user existed on prod, so minted a throwaway
`test_account` user (id 32, `referred_by_id=9`) directly via SQL, plus a throwaway admin `pc_sessions`
token (admin id 4, matching the [[sessions/2026-07-17]]/[[sessions/2026-07-15]] precedent of hitting
the real HTTP endpoint rather than guessing at behavior). PATCH `/api/admin/users/32`
`{"tier_status":"active"}` (required a real `Origin` header to pass the CSRF check, curl sends none by
default): `pc_referral_credits` got `(referrer_id=9, referee_id=32, months_granted=2, granted_at=now)`;
referrer's `tier_renews_at` moved `NULL → NOW()+2mo` (correct baseline, no prior date); referrer's
`tier_status` stayed `active` (already was); one `referral_reward_granted` audit row landed. Re-PATCHed
referee to `past_due` then back to `active` — confirmed **no second grant** (same single credit row,
unchanged `tier_renews_at`, still exactly 1 audit row). `GET /api/me/referral-stats` as referrer id 9
returned `{"joined":0,"converted":1,"months_earned":2}` — correct. All test data (user, credit row,
both session tokens, audit row, `tier_renews_at` reset) cleaned up after; confirmed zero leftover rows.

`php -l` clean over SSH on all 6 touched backend files (no local PHP). Deployed via `bash deploy.sh
--all` (schema + code); smoke check passed. Full detail: [[wiki/analyses/2026-07-20-referral-months-migration]].
