User has direct, concrete operational knowledge of how vendor communication actually happens in this domain — e.g. WhatsApp auto-zipping multiple shared images into one download when a vendor shares a price list that way (this drove the whole design target for [[../wiki/analyses/2026-07-02-zip-upload-spec|the ZIP upload feature]] — it's for that specific case, not a general multi-page-document uploader), and real vendors using non-standard tiered-pricing breakpoints (a real file used 1/10/50-kit tiers, not the assumed 1/10/100).

They also catch real compound/product-identity errors by domain knowledge, not by auditing the data — e.g. spotted that "BPC 5mg+TB5mg combo" (a blend of BPC-157 + TB-500) had been extracted as a spec variant under the plain BPC-157 product ("this is not the same as BPC-157"), which was live on the comparison table mixing blend pricing into pure-compound rows. This is exactly the class of error [[../wiki/concepts/variant-compounds|the variant-compound watchlist]] exists to catch, caught here by the user's own knowledge of the compounds rather than by any watchlist automation.

This extends to catching wrong *formulas*, not just wrong categorization or thresholds: independently flagged that `price_per_unit` (computed everywhere as `price_usd / numeric_value`) was missing the kit-vial-count factor — the real formula is `price_usd / (kit_vial_count * numeric_value)`, since a "kit" is multiple vials sold for one price. This had been wrong since the feature was built, silently overstating $/mg on every price where `kit_vial_count != 1` — a bug in the app's actual core metric (the whole comparison table is built around $/unit), caught by domain reasoning about what a "kit" physically is, not by testing or code review.

**Why:** Proposed a padded/generic default for the ZIP entry cap (5, "a little headroom over the typical case"). User corrected it down to 3 with a specific real-world reason: "most of the time the zip is just because they sent over 3 or 4 images." Their number was more specific and better-targeted than my generic safety-margin instinct. The BPC-157/TB-500 catch and the price_per_unit formula catch both show this domain expertise extends well beyond operational logistics into actual compound identity and pricing-math correctness — three separate instances now of them catching something a code-level review wouldn't have.

**How to apply:** When the user gives a concrete number or behavior driven by real operational experience, prefer it over a generically-padded engineering default. If a proposed number/threshold isn't obviously grounded in something concrete, it's worth surfacing the reasoning (or asking what's driving it) rather than assuming my own padding instinct is the one to bias toward. When they flag that two named compounds/products aren't equivalent, or that a formula doesn't match the real-world physical setup (kits, vials, tiers), treat it as a real, actionable bug immediately — don't second-guess it against what the code currently does, and consider proactively checking for the same failure shape elsewhere rather than only fixing the one instance they named.
