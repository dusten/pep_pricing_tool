41. **Slow-query capture: `min_examined_row_limit` in my.cnf (future, only if needed)** — deliberately deferred while shipping the 2026-07-10 slow-query-log fix (migration 026, backlog #7). That fix filters at the *import* layer — the hourly event only pulls genuinely-slow/heavy rows out of `mysql.slow_log` into `pc_slow_query_cache`, so the admin-facing table stays clean. But MySQL still *writes* every no-index/tiny-table query into `mysql.slow_log` first (they're just wiped hourly by the event's `DELETE FROM mysql.slow_log WHERE db='tmgb_price'`). Setting `min_examined_row_limit` (e.g. ~1000) in `/etc/my.cnf.d/tmgb-perf.cnf` would stop MySQL logging those tiny scans at the source, cutting the hourly event's scan work. **Not done because** it's a **server-wide** setting on a box shared with the grp app (`themightygroupbuy`) — it would change grp's slow-log behavior too, and needs a MySQL restart (or `SET GLOBAL`) to apply. Only worth doing if `mysql.slow_log` volume becomes a real problem (it isn't at current scale — the hourly wipe keeps it small). If pursued: coordinate with the grp app owner, set it in my.cnf for persistence, and confirm both apps' slow-query events still capture what they need.

- **Vendor upload dedup, two layers (2026-07-03)** — backlog #14, expanded past its original file-hash-only framing after discussion surfaced that byte-hash alone misses "different file, same data" re-uploads:
