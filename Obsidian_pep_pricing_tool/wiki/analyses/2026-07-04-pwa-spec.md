# PWA Support — Full Spec

User surfaced that the app was originally meant to be installable as a Progressive Web App on iOS, and asked for a spec to implement once icon assets are ready (user is generating icons separately, following the navy `#0E2245` / gold `#C8A227` design system). This covers the **full** offline-capable PWA, not just bare installability — user's explicit choice.

## Current state (confirmed by inspection, not assumed)

Nothing PWA-related exists today:
- No `manifest.json` anywhere in the repo.
- No icon files at all — `index.html`'s `<link rel="icon" href="/favicon.svg">` points at a file that has never existed.
- No service worker, no `vite-plugin-pwa`, no Workbox.
- Confirmed live: `curl https://price.themightygroupbuy.com/favicon.svg` and `/manifest.json` both return **HTTP 200 containing the SPA's `index.html`**, not a real asset or a 404. This isn't a server misconfiguration — `public/.htaccess` rule 1 (`RewriteCond %{REQUEST_FILENAME} -f` → serve directly) already serves any *real* file at that path without touching PHP; these paths just don't have real files yet, so rule 3 (catch-all → `index.php`) swallows them instead.

**Implication for implementation: no Apache/vhost changes are needed.** Once real files exist at `public/manifest.json`, `public/favicon.svg`, `public/icons/*.png`, etc., `.htaccess` rule 1 serves them directly and correctly. This was verified against the actual deployed `public/.htaccess`, not assumed.

## 1. Icon assets needed (user is producing these separately)

| File | Size | Purpose |
|---|---|---|
| `public/favicon.svg` | vector | Modern browsers, tab icon |
| `public/favicon-32.png` | 32×32 | Legacy fallback |
| `public/apple-touch-icon.png` | 180×180 | iOS home-screen icon (iOS ignores the manifest's icons on older versions — this file is load-bearing for iOS specifically) |
| `public/icons/icon-192.png` | 192×192 | Manifest `icons[]`, Android home screen / task switcher |
| `public/icons/icon-512.png` | 512×512 | Manifest `icons[]`, splash generation, app stores if ever wrapped |
| `public/icons/icon-maskable-512.png` | 512×512 | Manifest `icons[]` with `"purpose": "maskable"` — needs safe-zone padding (Android adaptive-icon masking crops a circle/squircle out of the image; a tight edge-to-edge icon gets clipped). Keep the mark inside the center ~80% of the canvas. |

Background should be solid `#0E2245` (navy) for all icons — no transparency, since iOS renders transparent PNG icons on an ugly default background.

## 2. `public/manifest.json`

```json
{
  "name": "TheMightyGroupBuy — Price Comparison",
  "short_name": "TMGB Price",
  "description": "Peptide vendor price comparison — TheMightyGroupBuy",
  "start_url": "/dashboard",
  "scope": "/",
  "display": "standalone",
  "orientation": "portrait",
  "background_color": "#F2EBD9",
  "theme_color": "#0E2245",
  "icons": [
    { "src": "/icons/icon-192.png", "sizes": "192x192", "type": "image/png", "purpose": "any" },
    { "src": "/icons/icon-512.png", "sizes": "512x512", "type": "image/png", "purpose": "any" },
    { "src": "/icons/icon-maskable-512.png", "sizes": "512x512", "type": "image/png", "purpose": "maskable" }
  ]
}
```

Notes:
- `start_url: "/dashboard"` rather than `/` — opening the installed app should land a logged-in user somewhere useful; `DashboardView.vue` already exists and the router already redirects unauthenticated visits appropriately, so this doesn't need new routing logic.
- `background_color` (cream `#F2EBD9`, the app's own `--background` token) is what shows briefly during launch before the first paint — should match the app background, not the navy primary, so the transition into the real UI isn't jarring.
- `theme_color` (navy `#0E2245`) tints the iOS status bar / Android toolbar.

## 3. `frontend/index.html` additions

```html
<link rel="manifest" href="/manifest.json" />
<link rel="apple-touch-icon" href="/apple-touch-icon.png" />
<meta name="theme-color" content="#0E2245" />

<!-- iOS-specific: Safari does not read display/orientation from the
     manifest reliably on all versions — these are the actual load-bearing
     tags for "opens full-screen, no browser chrome" on iOS. -->
<meta name="apple-mobile-web-app-capable" content="yes" />
<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent" />
<meta name="apple-mobile-web-app-title" content="TMGB Price" />
```

`black-translucent` lets the app's own navy header sit under the status bar rather than showing a plain black or white bar — matches the existing navy top bar (`--topbar-bg`/`--sidebar-bg`).

**Optional, not core scope:** iOS splash screens (`<link rel="apple-touch-startup-image">`) need one image per device/orientation size (a dozen+ variants across current iPhone/iPad models) since iOS doesn't auto-generate them from the icon. Skip unless the blank-white-flash on launch becomes an actual complaint — a tool like `pwa-asset-generator` can batch-produce them later from the 512px icon without new design work.

## 4. Service worker (full offline support, per user's chosen scope)

### Build tooling decision — flag for the user at implementation time

Two real options, genuinely different tradeoffs:

- **`vite-plugin-pwa`** (new frontend devDependency, wraps Workbox) — auto-generates the service worker from the Vite build manifest, handles cache versioning/invalidation correctly out of the box (the single easiest thing to get wrong by hand), auto-injects the `<link rel="manifest">` and can even generate icon variants from one source image. Standard, well-trodden path for a Vite SPA specifically.
- **Hand-rolled `sw.js`** — zero new dependencies (matches this project's general preference for minimal footprint / no-Composer-equivalent-on-the-frontend instinct), but the caching/versioning logic below has to be maintained by hand and is easy to get subtly wrong (stale JS bundle served forever is the classic failure mode).

Given how often this app deploys (this session alone shipped ~8 live deploys), **`vite-plugin-pwa` is the safer recommendation** — but noting the hand-rolled path stays open since it's a genuine, defensible choice for this codebase's stated dependency-minimalism preference. Pick at implementation time.

### Caching strategy (applies to either approach)

The critical constraint: **this app deploys frequently and data must never be served stale from cache.** Get the strategy backwards and a subscriber's Comparison table silently shows week-old prices with no error.

| Resource | Strategy | Why |
|---|---|---|
| `/api/*` (all backend calls) | **Network-only, never cached** | Pricing/cart/auth data must always be live. No exceptions — this is the one rule that can't bend. |
| Built JS/CSS (`/dist/assets/*`, content-hashed filenames) | **Cache-first, keyed by the hashed filename** | Vite already gives every build a new hash per file — safe to cache aggressively forever, since a new deploy naturally produces new URLs, never a stale-content collision. |
| `index.html` / the SPA shell | **Network-first, falling back to cache when offline** | This is the one HTML document that must reflect the *latest* build's asset references — never let a cached shell point at asset hashes from an old deploy. |
| `manifest.json`, icons | **Cache-first** | Static, rarely changes. |

### Update/refresh UX

A service worker that silently updates in the background but never tells the user is its own footgun (they keep using stale JS until they happen to fully close and reopen the app). Minimum viable UX: on detecting a new service worker version is waiting, show a small "Update available — tap to refresh" banner (a single `AppLayout.vue` addition, reusing the app's existing banner/pill visual language) rather than force-reloading without warning.

### Registration

`frontend/src/main.js` registers the service worker after the Vue app mounts (not blocking initial paint):
```js
if ('serviceWorker' in navigator) {
  window.addEventListener('load', () => navigator.serviceWorker.register('/sw.js'))
}
```
(If `vite-plugin-pwa` is chosen, its `virtual:pwa-register` helper replaces this and additionally exposes the "update available" callback needed for the banner above.)

## 5. iOS-specific caveats worth knowing before implementing

- iOS Safari's service worker support has real limits vs. desktop Chrome: no periodic background sync, no push notifications without extra Apple-specific setup, and Safari can evict an installed PWA's cache storage more aggressively under storage pressure. None of this blocks the "offline app-shell" goal above, but don't assume iOS behaves identically to Android/Chrome once implemented — test on a real iPhone, not just desktop Safari's responsive mode.
- "Add to Home Screen" on iOS is a manual user action (Share sheet → Add to Home Screen) — there's no Chrome-style automatic install prompt on iOS. Nothing to build for this; just worth knowing so no one goes looking for an "install banner" on iPhone.
- Re-confirm `start_url` and `scope` behave as expected once real icons exist — iOS is known to be pickier than Android about honoring `display: standalone` if the manifest has any malformed field.

## 6. Testing checklist (for whoever implements)

- [ ] `manifest.json` and all icon files return real content (not the SPA shell) via direct `curl` — confirms `.htaccess` rule 1 is serving them.
- [ ] Chrome DevTools → Application → Manifest shows no errors, icons render in the panel.
- [ ] Lighthouse PWA audit passes installability checks.
- [ ] Real iPhone: Safari → Share → Add to Home Screen → icon looks correct, launches standalone (no browser chrome), status bar color matches navy theme.
- [ ] Deploy a real code change after installing the PWA, confirm the "update available" banner appears and refreshing picks up the new build (not stuck on stale JS).
- [ ] Airplane mode: app shell still opens (even if API calls correctly fail/show an offline state) — confirms the cache-first app-shell strategy actually works, not just the network-only API rule.

## Non-goals (explicitly out of scope for this spec)

- Push notifications (separate, much bigger scope — Apple-specific web push setup, backend infrastructure, user opt-in UX).
- Background sync / offline write queueing (e.g. adding to cart while offline, syncing later) — this app's core value is live pricing data; offline writes don't make sense for this product.
- iOS splash screen images — noted above as optional, deferred until it's an actual complaint.
